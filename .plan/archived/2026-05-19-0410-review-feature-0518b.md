## 重要指摘事項（優先度順）

### [High] 全コマンドの stdout にバージョンバナーが出力される

- **指摘者:** System Architect, Maintainability Guardian, Legacy Bridge Architect, UI Expert, Domain Logic Expert, API & Contract Negotiator, DX Advocate（延べ12名のエージェントが言及）
- **場所:** `src/main.rs:191`
- **影響:** `sasurahime scan | grep uv` や `sasurahime --yes` を cron/CI に組み込むと、出力先頭にバージョン文字列が混入する。JSONパースやgrepパターンマッチを破壊する。多くの CLI ツール（cargo, rustc, docker 等）は通常起動時にバージョンを表示しない。
- **対処:** バージョンバナーを stdout ではなく stderr に出力する（例: `eprintln!`）。または `--quiet` フラグを追加して抑制可能にする。

### [Medium] `main.rs` の CleanTarget match arm 14個が同一構造で重複

- **指摘者:** System Architect, Maintainability Guardian, Refactoring Evangelist
- **場所:** `src/main.rs:222-361`
- **影響:** 新しいターゲットを追加するたびに14箇所の変更が必要になる。「Cleaner を生成して with_spinner で包んで freed を表示」のパターンが全く同じ構造で繰り返されている。
- **対処:** マクロまたは共通関数に抽出する（例: `run_cleaner(name, || cleaner.clean(dry_run))`）。System Architect の指摘が最も包括的。

### [Medium] Release profile の LTO + codegen-units=1 による CI ビルド時間増加

- **指摘者:** Tuning Expert, SRE/Ops Specialist, FinOps Consultant, DX Advocate
- **場所:** `Cargo.toml:21-25`
- **影響:** `lto = true` + `codegen-units = 1` は release ビルド時間を 3〜5 倍に引き上げる。GitHub Actions 等で配布用アーティファクトを毎回ビルドする場合、CI コストに直結する。
- **対処:** 分割プロファイル戦略を検討（`[profile.ci]` では簡易最適化、配布のみフル LTO）。または `profile-overrides` で依存関係のみ LTO を効かせる。

### [Medium] `interactive.rs` に TOCTOU race condition

- **指摘者:** Data Integrity Expert
- **場所:** `src/interactive.rs`
- **影響:** `detect()` で削除可能サイズを計算してから `clean()` が実行されるまでの間に、ファイルが追加/削除される可能性がある。実際の解放量が表示とずれる。
- **対処:** 設計上の制約としてドキュメントに記載。深刻なデータ破壊リスクではないが、認識は必要。

### [Medium] Cargo.toml に license フィールド未定義 + 依存関係のバージョン混在

- **指摘者:** Supply Chain & Dependency Sentinel
- **場所:** `Cargo.toml`
- **影響:** `license` フィールドがなく SBOM 生成ツールがライセンス情報を取得できない。`windows-sys` が3バージョン混在。
- **対処:** `license = "MIT"` を追加。`cargo update` で重複バージョンの解消を試みる。
